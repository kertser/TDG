# TDG — Платформа тактических командно-штабных учений

> 🌐 [English](README.md) | **Русский**

Многопользовательская веб-платформа для проведения тактических и командно-штабных учений с ИИ-противником, совместным нанесением обстановки на карту, анализом местности и распознаванием приказов на естественном языке.

## Возможности

- **Интерактивная тактическая карта** — карта на основе Leaflet с военными символами MIL-STD-2525D (milsymbol.js), масштабируемые маркеры, сетка с рекурсивным «улиточным» дроблением, высотные отметки (▲) с цифрами высот
- **Туман войны** — серверная фильтрация видимости через PostGIS `ST_DWithin` + учёт рельефа при расчёте секторов обзора; игроки видят противника только в пределах прямой видимости; разведчики и снайперы в режиме маскировки практически незаметны; тип и эшелон вражеских подразделений скрыты (отображается только общая категория)
- **Сектор обзора (LOS)** — видимость рассчитывается лучевым методом (raycast) вместо простых окружностей; лесные массивы и строения перекрывают обзор; высота глаз зависит от типа подразделения; модель поглощения видимости
- **Тактический поиск пути A\*** — маршруты движения с учётом рельефа на ячейках глубины 1 (~333 м); учитывает стоимость местности, уклон, минные поля, опасные зоны, укрытия и близость к своим; режим скорости влияет на маршрут (медленно = скрытность, быстро = скорость); путевые точки вычисляются сразу при получении приказа и пересчитываются каждые 5 тактов; сглаживание сплайном Катмулла-Рома на фронтенде
- **Авиация и воздушно-десантные операции** — 3 типа авиации (ударный вертолёт, транспортный вертолёт, разведывательный БПЛА) с возможностью игнорировать рельеф; специальные типы приказов (воздушный десант, эвакуация раненых, авиаудар); большая высота наблюдения (100–200 м); двуязычные ключевые слова EN/RU
- **Совместные условные знаки** — инструменты рисования в реальном времени (стрелки, ломаные линии, прямоугольники, маркеры, эллипсы, измерение расстояний) через WebSocket; видны всем сторонам
- **Анализ местности** — автоматическая классификация по данным OSM Overpass + ESA WorldCover + Open-Elevation; таксономия из 12 типов местности с тактическими модификаторами; ручная окраска от администратора; потоковый прогресс через SSE; обнаружение высотных отметок
- **Механизм правил** — детерминированная тактовая симуляция: движение (скорости slow/fast для каждого типа подразделения с A\*-маршрутизацией), обнаружение (на основе LOS с маскировкой разведчиков), бой (прямой + площадной огонь с ограниченными залпами, координация боевых ролей, **преимущество обороняющегося**), моральный дух, подавление (восстановление с учётом позиции), боеприпасы, связь, выход из боя, инженерное оборудование позиций, отдых и восстановление, пополнение запасов, действия инженерных подразделений и целенаправленная поддержка тылом
- **Преимущество обороняющегося** — при равных силах обороняющийся всегда в выигрыше (соотношение потерь 1,77:1). Атакующие теряют −35% огневой эффективности; обороняющиеся получают +15%. Восстановление после подавления у неподвижных защитников в два раза быстрее. Фланговый удар снижает защиту на 25%. При подавлении противника свыше 60% открывается «окно прорыва» — тактика «подавление → штурм» вознаграждается.
- **Координация боевых ролей** — подразделения, атакующие одну цель, автоматически распределяют роли: подавление (~40%, прикрывающий огонь с дистанции), штурм (1–2 пехотных подразделения сближаются), обход (атака под углом 60° через укрытую местность). Роли объявляются по радио.
- **Координация артиллерии и пехоты** — трёхуровневая защита от поражения своих: проактивный запрос прекращения огня на 250 м, автоматическая остановка «опасно близко» на 50 м, проверка при площадном огне. Артиллерия поддерживает как атакующие, так и обороняющиеся подразделения. Система явных запросов огня. Автоматический запрос артиллерийской поддержки при обнаружении цели.
- **Тактические объекты на карте** — статические объекты поля боя: колючая проволока, минные поля, окопы, заграждения, ДОТы, мосты, командные пункты, склады топлива, аэродромы (поворотные) и т.д. Маркеры в стиле НАТО с системой обнаружения по сторонам
- **Площадные эффекты** — временные полигональные угрозы: дым (блокирует обнаружение), туман (снижает видимость), пожар (наносит урон, блокирует движение), химическое облако (сильный урон пехоте). Все эффекты затухают со временем. Визуальные эффекты попаданий (взрывы).
- **Система снабжения** — склады боеприпасов (+10% в такт), подразделения логистики (мобильное +8% в такт), медицинские пункты (+1% силы в такт). Тип приказа «пополнение» с автоматическим движением к ближайшему источнику снабжения.
- **Система командования** — иерархическое дерево подразделений с контролем полномочий, назначением личного состава, редактированием иерархии методом drag-and-drop, разделением и слиянием, проверкой прав и поддержкой реорганизационных приказов
- **Панель администратора** — плавающее окно администратора с мастером создания сессии (4 шага: Настройка → Участники → Местность → Готово), режим «видит всё», панель подразделений, конструктор сценариев, редактор командной вертикали, управление анализом местности, редактор типов подразделений, журнал отладки, размещение площадных эффектов
- **Система приказов** — ввод приказов текстом с распознаванием через ИИ (GPT-4.1, двуязычный EN/RU), детерминированная интерпретация намерений, 3-уровневая маршрутизация с оптимизацией стоимости (ключевые слова → nano → полная модель), радиоответы подразделений с тактической оценкой, умное предложение построения, распознавание координат/сетки/«улиточных» адресов и высотных отметок, немедленное назначение задачи, обработка инженерных/логистических задач с учётом объектов карты и доктринальный разбор приказов на разделение/слияние и поддержку
- **Разговорник приказов** — управляемый данными двуязычный лексикон (`order_phrasebook.toml`) для классификации команд, обнаружения типов приказов, разбора скорости/построения, правил открытия огня, ссылок на местность и 60+ регрессионных тестов; загружается при старте сервера
- **Контекстно-зависимое промптирование** — тактическая доктрина загружается из `FIELD_MANUAL.md` и инжектируется в промпты по тематикам: огонь, разведка, инженеры, логистика, авиация, объекты карты, разделение/слияние
- **Сжатие промптов и поиск по контексту** — 4-слойная упаковка контекста: фрейм задачи, дельты состояния (не полная история), тематические карточки доктрины (BM25-подобный поиск), динамически выбранные few-shot примеры. Подавление пустых секций. Детерминированное разрешение непрерывности («та же цель», «тот мост»). Кэш результатов промптов с TTL 5 мин. Типичный размер промпта: 1292–1648 токенов.
- **Поддержка локального LLM** — развёртывание без доступа к сети через llama.cpp (OpenAI-совместимый API). Docker Compose профиль `llm` с настройками под CPU: ctx=4096, reasoning off, квантизация Q4_K_M, переиспользование KV-кэша. Настраивается через `LOCAL_MODEL_URL` / `LOCAL_MODEL_NAME` в `.env`. Три режима разбора: `llm_first` (по умолчанию), `keyword_first` (устаревший), `keyword_only` (офлайн).
- **Радиосвязь** — тактический радиоканал между командирами сессии с выбором адресата, тремя фильтрами каналов (Все / 💬 Чат / 📡 Подразделения) и индикатором непрочитанных. Авто-сообщения по радио: доклады о выполнении задачи, запросы поддержки, рапорты о потерях, обмен информацией при огневых задачах, координация атаки, остановка при обнаружении противника
- **Доклады** — пять автоматически формируемых типов: РАЗВЕДДОНЕСЕНИЕ (контакт с противником), ДОНЕСЕНИЕ ОБ ОБСТРЕЛЕ (под огнём), ДОНЕСЕНИЕ О ПОТЕРЯХ (уничтожение подразделения), ДОКЛАД ОБ ОБСТАНОВКЕ (периодический), РАЗВЕДЫВАТЕЛЬНАЯ СВОДКА (разведывательный итог). Двуязычные RU/EN. Значок непрочитанных на вкладке панели.
- **Воспроизведение сессии** — пошаговое воспроизведение с элементами управления (пуск/пауза, шаг вперёд/назад, скорость 0,5×–4×), ползунок временной шкалы, рендеринг позиций подразделений по тактам с плавной анимацией и разбором действий (AAR), генерируемым ИИ
- **Интернационализация (i18n)** — полное переключение языка интерфейса EN/RU через модуль `KI18n`; HTML-атрибуты `data-i18n` для декларативного перевода; выбор языка в настройках пользователя; перерендеринг в реальном времени при смене языка
- **Контроль задач и победные условия** — детерминированная проверка контроля территории каждый такт (`objective_control.py`): события «задача захвачена»/«задача оспаривается», обнаружение полного уничтожения, проверка порога `objectives_to_win` — всё без LLM. ИИ-рефери оценивает произвольные условия победы каждые 5 тактов. Поддержка лимита ходов. Автозавершение при победе или исчерпании ходов.
- **Красные ИИ-противники** — агенты-командиры с 4 доктринальными профилями (агрессивный/сбалансированный/осторожный/оборонительный), ограниченным знанием обстановки (без утечки синих данных), решениями LLM с резервным правилам-based-fallback
- **Инжекция трудностей тренером** — администратор может ввести осложнения для любого подразделения в ходе учений: `breakdown` (поломка), `comms_failure` (потеря связи), `position_error` (ошибка местоположения), `ammo_shortage` (нехватка боеприпасов), `fuel_depletion` (истощение топлива), `commander_casualty` (потеря командира) — каждый с длительностью (такты) и величиной. Хранится в `unit.capabilities.friction`. Снять вручную или дождаться истечения.
- **Адаптивное обучение разговорника** — статистический майнинг предложений по расширению разговорника приказов из реальных данных сессий. Межсессионная кластеризация (требуется ≥5 сессий, ≥3 пользователя), опциональный ИИ-судья качества, рабочий процесс проверки человеком (одобрить/отклонить/применить). Одобренные записи записываются в `order_phrasebook.toml` с горячей перезагрузкой без перезапуска.
- **Интерактивный обучающий тур** — пошаговый онбординг-тур с подсветкой элементов (spotlight) автоматически запускается при первом входе (`KTutorial`). Прохождение сохраняется на сервере для каждого пользователя; повторный запуск из настроек.
- **Журнал событий** — хронологический список событий, панель докладов с фильтрацией по каналам, журнал приложения (отдельно от тактических данных)
- **Редактируемая конфигурация** — определения типов подразделений и константы отображения хранятся в JSON-конфигах (`unit_types.json`, `units_config.json`) вместо захардкоженного JavaScript

## Быстрый старт

### Развёртывание в продакшен (рекомендуется)

**Одна команда для полного развёртывания** — самый простой способ запустить платформу.

**Windows (PowerShell)**:
```powershell
# 1. Клонировать и настроить
git clone <repository-url>
cd KShU
Copy-Item .env.example .env
# Отредактировать .env: задать OPENAI_API_KEY, SECRET_KEY, ADMIN_PASSWORD

# 2. Развернуть всё (PostgreSQL + Redis + Backend + Nginx + Локальный LLM)
.\deploy.ps1

# 3. Открыть
# Интерфейс:    http://localhost
# Документация API: http://localhost/api/docs
# Локальный LLM:    http://localhost:8081
```

**Linux/Unix**:
```bash
git clone <repository-url>
cd KShU
cp .env.example .env
# Отредактировать .env: задать OPENAI_API_KEY, SECRET_KEY, ADMIN_PASSWORD
chmod +x deploy.sh
./deploy.sh
```

### Флаги скрипта развёртывания

`deploy.ps1` (Windows) и `deploy.sh` (Linux) поддерживают одинаковые флаги:

| Флаг | Описание |
|---|---|
| *(нет)* | Собрать образы (из кэша) и запустить весь стек |
| `--rebuild` | Остановить, пересобрать образы с `--no-cache`, перезапустить. **База данных сохраняется.** |
| `--clean` | **Полная очистка**: удалить контейнеры, **тома (данные будут удалены)**, образы, кэш сборки — пересобрать с нуля |
| `--down` | Остановить и удалить все контейнеры. **Том базы данных сохраняется** — данные выживают. |
| `--logs` | Слежение за логами контейнеров после запуска (`Ctrl+C` для выхода) |

```powershell
# Примеры для Windows
.\deploy.ps1 --rebuild        # принудительная пересборка
.\deploy.ps1 --rebuild --logs # пересборка, затем слежение за логами
.\deploy.ps1 --clean          # ядерный вариант
.\deploy.ps1 --down           # остановить всё
```

> **Отключить локальный LLM**: закомментировать `COMPOSE_PROFILES=llm` в `.env`.

## Архитектура развёртывания

```
Браузер клиента (http://localhost)
        ↓
    Nginx (порт 80)
    ├─→ Статические файлы фронтенда
    ├─→ /api/*  → Backend:8000
    └─→ /ws/*   → Backend:8000 (WebSocket)
        ↓
    FastAPI Backend (порт 8000)
    ├─→ PostgreSQL:5432 (+ PostGIS)
    ├─→ Redis:6379 (pub/sub + кэш)
    └─→ LLM:8081 (локальный llama.cpp)
```

| Сервис | Порт | Описание |
|---|---|---|
| `nginx` | 80 | Обратный прокси + статические файлы фронтенда |
| `backend` | 8000 | FastAPI приложение + автомиграции (внутренний) |
| `postgres` | 5432 | PostgreSQL 16 + PostGIS 3.4 |
| `redis` | 6379 | Redis (pub/sub, кэш сессий) |
| `llm` | 8081 | llama.cpp с OpenAI-совместимым API |

У всех сервисов есть проверки работоспособности, политики перезапуска и правильные цепочки зависимостей.

## Настройка окружения

Скопировать `.env.example` в `.env`. Обязательные параметры:

```env
OPENAI_API_KEY=sk-...         # для разбора приказов LLM и Красного ИИ
SECRET_KEY=<random-64-chars>  # для JWT-токенов
ADMIN_PASSWORD=<strong-pass>  # для панели администратора
```

Опциональные параметры:
```env
LOCAL_MODEL_URL=http://localhost:8081/v1
LOCAL_MODEL_NAME=local
LOCAL_TRIAGE_ENABLED=true
LLM_PARSING_MODE=llm_first    # llm_first | keyword_first | keyword_only

OPENAI_MODEL=gpt-4.1
OPENAI_MODEL_MINI=gpt-4.1-mini
OPENAI_MODEL_NANO=gpt-4o-mini
```

Сгенерировать `SECRET_KEY`:
```powershell
# Windows PowerShell
[Convert]::ToBase64String((1..48 | ForEach-Object { [byte](Get-Random -Max 256) }))
```

## Логи, мониторинг и устранение неисправностей

```powershell
# Логи всех сервисов
docker compose logs -f

# Конкретный сервис
docker compose logs -f backend
docker compose logs -f llm

# Состояние сервисов
docker compose ps
curl http://localhost/health
curl http://localhost:8081/health
```

**Backend не запускается** — проверить `docker compose logs backend`. Частые причины: отсутствует `OPENAI_API_KEY`, база не готова (подождать 30 с, backend повторяет попытки автоматически), порт 8000 уже занят.

**Миграции базы данных не выполняются**:
```powershell
docker compose exec backend alembic upgrade head
# или полный сброс (данные будут уничтожены):
.\deploy.ps1 --down
docker volume rm tdg_pgdata
.\deploy.ps1
```

**Nginx 502 Bad Gateway** — backend ещё не прошёл health check: `docker compose restart backend`

**Локальный LLM не отвечает**:
```powershell
docker compose logs llm
# Прогрев модели после первого запуска (долгая первая загрузка):
python scripts\warm_local_llm.py
```

## Сохранность данных и сброс

Данные игры хранятся в томе Docker `pgdata`. Они **выживают** при `--down` и подключаются при следующем запуске.

Полный сброс (очистка базы):

```powershell
# Windows — удаляет всё включая данные
.\deploy.ps1 --clean

# Linux/Unix
./deploy.sh --clean
```

Резервное копирование перед сбросом:
```powershell
docker compose exec postgres pg_dump -U tdg tdg > backup.sql
```

## Усиление безопасности для продакшен

1. **Сменить все пароли по умолчанию** в `.env` — `SECRET_KEY`, `ADMIN_PASSWORD` и пароли базы данных в `docker-compose.yml`
2. **Включить HTTPS** — использовать SSL-конфигурацию nginx или обратный прокси (Traefik, Caddy)
3. **Ограничения ресурсов** — добавить в `docker-compose.yml` для сервиса backend:
   ```yaml
   deploy:
     resources:
       limits:
         cpus: '2'
         memory: 4G
   ```
4. **Регулярное резервное копирование** тома `pgdata`
5. **Мониторинг** — Prometheus + Grafana на эндпоинтах `/health`

### Разработческое окружение (альтернатива)

Для активной разработки с горячей перезагрузкой:

#### 1. Требования
- Python 3.12+
- Docker & Docker Compose

#### 2. Запустить инфраструктуру
```powershell
docker compose up -d
```
Запускает PostgreSQL + PostGIS (порт 5432) и Redis (порт 6379).

> **Примечание:** При обновлении с предыдущей версии, где DB называлась `kshu`, сначала выполнить `docker compose down -v`, затем `docker compose up -d`.

#### 3. Установить Python-зависимости
```powershell
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

#### 4. Настроить окружение
```powershell
Copy-Item .env.example .env
# Отредактировать .env, задать OPENAI_API_KEY
```

## Загрузка доктрины

- `FIELD_MANUAL.md` — авторитетный тактический источник.
- `backend/prompts/tactical_doctrine.py` загружает:
  - полные маркеры доктрины для глубокого рассуждения ИИ
  - краткие маркеры доктрины для компактного контекста разбора
  - тематические фрагменты (`DOCTRINE:TOPIC:*`) — промпты получают только релевантный тактический контекст
- Тактические регрессии следует расширять в виде пакетов сценариев, а не только изолированных тестов парсера. Поддерживать двуязычные RU/EN кейсы для манёвра, огня, инженерии, логистики, авиации, разделения/слияния и взаимодействия с объектами карты.
- Парсер приказов выбирает доктрину по семейству команд:
  - `fires`
  - `recon`
  - `engineers`
  - `logistics`
  - `aviation`

## Разговорник приказов

Парсер ключевых слов управляется файлом `backend/data/order_phrasebook.toml` — структурированным TOML, содержащим:

- **Лексикон классификации** — двуязычные ключевые слова команд/подтверждений/докладов/запросов статуса
- **Шаблоны обнаружения приказов** — готовность, координация, запросы огня, прорыв, минирование, разворачивание моста, строительство, дым, разделение/слияние, воздушная мобильность, прикрытие, отход, выход из боя, пополнение и др.
- **Ключевые слова скорости** — квалификаторы медленного/быстрого движения EN и RU (30+ слов каждый)
- **Шаблоны построений** — колонна, цепь, клин, «вее», уступ, ромб, каре, уступная колонна, «ёлочка» с явными префиксными шаблонами
- **Правила открытия огня** — огонь запрещён, огонь по готовности, только ответный огонь
- **Шаблоны объектов местности** — минные поля, колючая проволока, мосты, ДОТы, командные пункты, склады и т.д.
- **60+ регрессионных тестов** — записи `[[case]]` с ожидаемой классификацией, типом приказа, ссылками на местность, скоростью и типом объекта карты; проверяются тестовым набором

Разговорник загружается при запуске через `backend/services/order_phrasebook.py` и используется парсером для детерминированного сопоставления ключевых слов до любого обращения к LLM.

## Использование

1. Введите позывной и пароль, нажмите **Зарегистрироваться** (первый раз) или **Войти**
2. При первом входе запустится **интерактивный обучающий тур** — пошаговое знакомство с интерфейсом; можно закрыть в любой момент или перезапустить из настроек пользователя
3. Нажмите на сессию из списка для входа (сессии создаются администратором)
4. Нажмите **Запустить сессию** для инициализации подразделений из сценария
5. На карте появятся военные символы подразделений (с фильтрацией тумана войны для вашей стороны)
6. Используйте **панель управления картой** (верхний правый угол) для переключения инструментов рисования, сетки, подразделений, наложений, контактов, подписей и местности
7. **Рисовать наложения**: выбрать инструмент (стрелка, ломаная линия, прямоугольник, маркер, эллипс, измерение) и нанести на карту; синхронизируется в реальном времени через WebSocket
8. **Управлять подразделениями**: левый клик — выбрать, Shift+клик — добавить к выборке, удерживание левой кнопки — прямоугольная выборка, Alt+клик — переключаться между подразделениями в стопке; правый клик — контекстное меню (движение медленно 🐢/быстро ⚡, построение, разделить, слить, переименовать, назначить)
9. **Отдавать приказы** во вкладке **📡 Приказы** нижней командной панели (сначала выбрать подразделения или нажать **👥 Все**)
10. **Радиосвязь** во вкладке **📻 Радио** — отправлять тактические сообщения конкретным командирам или всем; фильтровать по каналу (Все / 💬 Чат / 📡 Подразделения)
11. **Продвигать симуляцию** нажатием **Выполнить приказы** — подразделения двигаются по A\*-оптимальным маршрутам, обнаруживают противника, ведут бой с координацией ролей и докладывают по радио
12. Просматривать события и доклады на вкладках панели (**События**, **Доклады** со значком непрочитанных); нажать **📋 имя сессии** для просмотра briefing сценария
13. Ссылаться на **высотные отметки** в приказах: *«Выдвинуться к высоте 170»* / *"Move toward height 170"*
14. **Сменить язык**: открыть настройки пользователя и выбрать English или Русский — весь интерфейс обновится в реальном времени
15. **Воспроизвести сессию**: навести на игровые часы (нижний правый угол) и нажать **Воспроизведение**; использовать элементы управления для пошагового просмотра или авто-воспроизведения; нажать **📊 ААР** для генерации разбора действий от ИИ

### Панель администратора
Нажмите кнопку администратора (🔑) и введите пароль:
- **Сессия** — управление запуском/паузой/тактом, мастер создания сессии (4 шага), удаление всех подразделений, сброс сессии
- **Монитор** — «видит всё» (обе стороны), панель подразделений с фокусом/редактированием/удалением/разделением/слиянием, переключатель журнала отладки (детальные данные движка по тактам), **инжекция трудностей тренером** (ухудшить любое подразделение в ходе учений: поломка, потеря связи, нехватка боеприпасов и др.)
- **Конструктор** — интерактивный конструктор сценариев с размещением подразделений кликом на карте, настройкой сетки, сохранением/загрузкой, сохранением сессии → сценарий
- **КВ** — полный редактор командной вертикали с drag-and-drop переназначением, массовым назначением/снятием
- **Пользователи** — управление участниками сессии
- **Типы** — редактор типов подразделений с живым SIDC-превью (скорости, дальности, личный состав, высота глаз)
- **Местность** — анализ местности (OSM + ESA + высоты), ручная окраска ячеек, очистка/перезагрузка
- **Эффекты** — размещение площадных эффектов: дым, туман, пожар, химические облака (временные полигональные угрозы)
- **Обучение** — запуск майнинга разговорника из данных о приказах сессии; просмотр и одобрение/отклонение кандидатур на расширение разговорника; применение одобренных записей в `order_phrasebook.toml` с горячей перезагрузкой

### Тактические объекты (препятствия и сооружения)
При открытой панели администратора:
- Размещать препятствия (колючая проволока, минные поля, окопы, противотанковые рвы, «зубы дракона») и сооружения (ДОТы, мосты, командные пункты, склады топлива, аэродромы, наблюдательные вышки и т.д.)
- **Поворот аэродрома**: перетащить оранжевый маркер ↻ на конце ВПП
- Правый клик на объект — контекстное меню: активировать/деактивировать, переключить обнаружение для Синих/Красных, удалить
- Перетаскивать объекты для перепозиционирования (центральный маркер для точек/аэродромов, центроидный маркер для линий/полигонов)
- У объектов есть обнаружение по сторонам: препятствия скрыты по умолчанию, открываются, когда LOS подразделения достигает их

### Анализ местности
Из вкладки Местность панели администратора:
1. Выбрать глубину анализа (1–4, глубина 2 ≈ ячейки 111 м)
2. Нажать **Анализировать** — прогресс поступает в реальном времени через SSE
3. Переключить видимость слоя местности из панели управления картой
4. Окрашивать отдельные ячейки инструментом рисования местности
5. Модификаторы местности (движение, видимость, защита, атака) поступают в движок симуляции
6. **Высотные отметки** (▲) определяются автоматически и отображаются на карте с цифрами высот
7. Граф поиска пути A\* строится и сохраняется автоматически после анализа местности

## Правила игры и симуляция

### Тактовая симуляция
Игра продвигается дискретными тактами (по умолчанию: 1 минута игрового времени за такт). Каждый такт обрабатывает:
1. **Красный ИИ** → агенты ИИ принимают решения для подразделений Красных
2. **Приказы** → проверенные приказы назначают задачи подразделениям; немедленное назначение при подтверждении
3. **Поиск пути** → путевые точки A\* вычисляются для всех движущихся подразделений (учёт рельефа, обход противника)
4. **Движение** → подразделения следуют путевым точкам A\* с учётом типа, рельефа, уклона, подавления, морального духа и погоды. Останавливаются перед обнаруженными минными полями и реками без мостов. При обнаружении противника в движении: остановка и запрос приказов.
5. **Обнаружение** → проверки видимости на основе LOS между противоборствующими сторонами; создание/обновление контактов; применение маскировки разведчиков
6. **Обнаружение объектов** → подразделения раскрывают скрытые препятствия/сооружения в зоне LOS
7. **Устаревшие контакты** → старые контакты затухают и в итоге удаляются
8. **Поддержка артиллерии** → простаивающая артиллерия автоматически назначается для поддержки атакующих или обороняющихся подразделений КВ. Явные запросы огня обрабатываются первыми. Координация прекращения огня с наступающей пехотой.
9. **Оборона** → прогресс инженерного оборудования для обороняющихся подразделений
10. **Ответный огонь** → подразделения под атакой автоматически атакуют ближайшего противника (кроме выходящих из боя)
11. **Бой** → атакующие подразделения ведут огонь с координацией ролей (подавление/штурм/обход). fire_effectiveness включает модификатор позиции (атакующий ×0,65, обороняющийся ×1,15). Урон, подавление — по огневой мощи, местности, боеприпасам. Фланговый удар снижает защиту защитника ×0,75. Площадной огонь — радиус поражения 150 м. Ограниченные залпы (по умолчанию 3). Опасно близко на 50 м.
12. **Восстановление от подавления** → не находящиеся под огнём подразделения восстанавливаются в зависимости от позиции: защитники/неподвижные 0,05/такт, атакующие/наступающие 0,02/такт
13. **Моральный дух** → подавление и потери снижают боевой дух; обороняющиеся +0,01/такт; атакующие под огнём −0,01/такт дополнительно; безопасность и близость к своим восстанавливают моральный дух; уничтожение противника повышает боевой дух поблизости; усталость от марша
14. **Связь** → сильное подавление может нарушить связь; подразделения офлайн продолжают последнюю задачу
15. **Боеприпасы и снабжение** → боеприпасы расходуются при каждом огневом такте; склады и подразделения логистики пополняют ближайших
16. **События и доклады** → значимые изменения состояния записываются; авто-доклады (РАЗВЕДДОНЕСЕНИЕ, ДОНЕСЕНИЕ ОБ ОБСТРЕЛЕ, ДОНЕСЕНИЕ О ПОТЕРЯХ, ДОКЛАД ОБ ОБСТАНОВКЕ, РАЗВЕДЫВАТЕЛЬНАЯ СВОДКА)
17. **Радиосвязь** → доклады простаивающих подразделений, запросы поддержки, рапорты о потерях, обмен при огневых задачах, координация атаки, сообщения при обнаружении противника в движении
18. **Площадные эффекты** → урон от огня/химического облака применяется; длительность тикается; истёкшие эффекты удаляются
19. **Проверка победы** → сначала детерминированная проверка контроля задач (уничтожение + порог, без LLM); затем LLM оценивает произвольные условия победы каждые 5 тактов; проверяется лимит ходов
20. **Рассылка** → обновлённое состояние отправляется всем подключённым клиентам через WebSocket

### Движение
- Каждый тип подразделения имеет уникальные скорости **медленно** (тактическое) и **быстро** (стремительное) в м/с
- **Авиация игнорирует весь рельеф** — вертолёты и БПЛА летят над водой, минными полями, препятствиями и крутым рельефом без штрафа
- **Тактический поиск пути A\***: наземные подразделения прокладывают маршруты с учётом рельефа, минных полей, наблюдения противника и непроходимой местности
- Режим скорости влияет на маршрутизацию: **медленно** предпочитает укрытые маршруты (лес, городская застройка), **быстро** предпочитает дороги и открытую местность
- `effective_speed = base_speed × terrain_factor × slope_factor × (1 - suppression × 0.7) × morale_factor × weather_mod`
- **Коэффициенты местности**: дорога=1,0, открытая=0,8, лес=0,5, городская=0,4, вода=0,05, поля=0,7, болото=0,3 и др.
- **Штраф за уклон**: `max(0,2, 1,0 - slope_deg/45)` — крутой рельеф резко замедляет движение
- Объекты карты влияют на движение: минные поля наносят урон/замедляют, колючая проволока замедляет и т.д.
- Наземные подразделения останавливаются перед обнаруженными минными полями (запрос инженеров) и у воды без мостов
- **Обнаружение при движении**: движущиеся (не атакующие) подразделения останавливаются при обнаружении противника и запрашивают приказы; возобновляют движение через 3 такта при отсутствии новых приказов

### Обнаружение и LOS
- **На основе сектора обзора**: 72 луча из позиции подразделения, препятствия (леса, здания) блокируют обзор
- **Высота глаз**: зависит от типа подразделения (наблюдательный пост=8 м, танки=3 м, пехота=2 м по умолчанию)
- **Вероятность обнаружения**: `base_prob × (1 - distance/range) × posture_mod × recon_bonus × concealment`
- **Маскировка разведки**: неподвижные разведывательные/снайперские/наблюдательные подразделения практически невидимы (макс. дальность обнаружения 300 м, базовая вероятность 10%, ограничение 25%)
- Детерминированный хэш обеспечивает воспроизводимость для replay

### Бой
- `fire_effectiveness = base_firepower × strength × ammo_factor × (1 - suppression) × terrain_mod × posture_mod`
- **Модификатор позиции** (преимущество обороны, историческое основание: правило НАТО 3:1, QJM Дюпюи):
  - Атакующее подразделение: ×**0,65** — наступление под давлением, нет заблаговременно подготовленного огня
  - Обороняющееся подразделение (авто-ответный огонь): ×**1,15** — заранее пристрелянные секторы, данные карточек стрельбы, знакомая местность
  - Артиллерия/подавляющий огонь: ×1,0 (без изменений)
  - Равные подразделения, открытая местность: **соотношение потерь 1,77:1** в пользу обороняющегося → нужно ~3:1 для прорыва
  - **Окно прорыва**: когда подавление противника ≥60%, штраф атаки линейно обнуляется — «подавить сначала, потом штурмовать» — правильная доктрина
- Высотное преимущество: +15% эффективности при стрельбе с более высокого рельефа
- **Координация боевых ролей**: несколько подразделений, атакующих одну цель, автоматически координируют — подавление (40%, прикрывающий огонь с дистанции), штурм (1–2 пехотных сближаются), обход (под углом 60° через укрытую местность)
- **Фланговый удар**: дополнительно ×0,75 к защите противника — подход с незащищённого фланга
- **Площадной огонь**: артиллерия/миномёты могут вести огонь по координатам сетки — радиус 150 м, урон убывает с расстоянием
- **Ограниченные залпы**: огневые задачи ограничены 3 залпами (настраивается), затем автозавершение
- **Опасно близко**: артиллерия автоматически прекращает огонь, если своё подразделение находится в 50 м от цели
- **Координация прекращения огня**: пехота, приближающаяся к зоне обстрела своей артиллерии (250 м), останавливается и запрашивает прекращение огня; артиллерия заканчивает последний залп, пехота возобновляет движение
- **Поддержка артиллерии**: автоматически назначается для поддержки как атакующих, так и обороняющихся подразделений КВ. Явные запросы огня перекрывают приказы готовности. Автозапрос при обнаружении цели.
- Подразделения с непрямым огнём (миномёты, артиллерия) имеют расширенную дальность, обозначенную пунктирными окружностями
- **Авто-ответный огонь**: подразделения под атакой без приказов атакуют ближайшего противника (кроме выходящих из боя)

### Типы подразделений
43 типа подразделений в `frontend/config/unit_types.json`, каждый с:
- Кодами SIDC MIL-STD-2525D (варианты Синих и Красных)
- Скоростями движения медленно/быстро (м/с)
- Дальностью обнаружения, дальностью огня, численностью личного состава
- Высотой глаз для расчётов LOS
- Флагом непрямого огня (миномёты, артиллерия)
- Специальными возможностями (авиационный обход рельефа, грузовместимость и т.д.)

**Подразделения ПВО** (4 типа):
- **ПЗРК (расчёт / отделение)** — 4–8 чел., дальность огня 3–3,5 км
- **ЗРК (отделение)** — 12 чел., дальность огня 8 км, обнаружение 10 км
- **Зенитная пушка (отделение)** — 10 чел., дальность огня 2,5 км

**Авиационные подразделения** (3 типа):
- **Ударный вертолёт** — 70 м/с быстро, обнаружение 5 км, огонь 4 км, высота глаз 100 м
- **Транспортный вертолёт** — 60 м/с быстро, обнаружение 3 км, грузоподъёмность 12 чел., высота глаз 150 м
- **Разведывательный БПЛА** — 35 м/с быстро, обнаружение 8 км, безоружный, высота глаз 200 м

Авиационные подразделения игнорируют все ограничения рельефа (вода, минные поля, препятствия, уклон) и управляются специальными типами приказов: `air_assault` (вертолётный десант), `casevac`/`medevac` (эвакуация раненых), `airstrike` (атакующий заход).

## Тестирование

### Тактические сценарные тесты
Автоматизированные тактические тесты проверяют поведение движка (движение, обнаружение, бой, координация):

```powershell
# Запустить все тактические сценарии (требуется запущенная инфраструктура backend)
python -m scripts.tactical_tests.run_all

# Запустчик тестов:
# 1. Создаёт тестовую сессию с подразделениями и приказами
# 2. Выполняет заданное количество тактов
# 3. Оценивает утверждения (события, обнаружения, состояние подразделений)
# 4. Генерирует HTML-отчёт: tactical_test_report.html
```

**10 тестовых сценариев**:
- Базовое движение (скорости по типу подразделения)
- Бронированный прорыв (комбинированные силы)
- Оборонительная позиция (окапывание, ответный огонь)
- Координация комбинированных сил
- Разведка инфильтрацией (маскировка)
- Встречный бой (взаимное обнаружение)
- Городской бой (эффекты рельефа)
- Ночные операции (модификаторы видимости)
- Преодоление водной преграды (требование мостов)
- Отход под давлением (моральный дух, выход из боя)

### Регрессия разговорника приказов
Файл `order_phrasebook.toml` содержит 60+ записей `[[case]]`, служащих регрессионными тестами парсера ключевых слов. Каждый случай задаёт входной текст и ожидаемые результаты (классификация, тип приказа, местоположения, скорость, тип объекта карты). Проверяются тестовым набором для предотвращения регрессий парсера.

Определения сценариев и фреймворк тестирования: `scripts/tactical_tests/`.

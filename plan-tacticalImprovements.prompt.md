# Plan: Тактические и игровые улучшения KShU

Система уже реализует богатую механику (LOS, A*, артиллерийская координация, инженерные задачи, красный ИИ, деза, авиация). Ниже — следующий слой: **углубление боевых механик**, **командный цикл**, **UX для тренировок**, **системные пробелы** и **адаптивное самообучение**.

---

## I. Боевые механики (глубина)

**1. Направление атаки в бою — использование `heading_deg`**
`heading_deg` хранится, но никак не влияет на урон. Добавить: расчёт угла между направлением атакующего и фронтом защитника → если атака с фланга (>60°) или с тыла (>120°) — автоматически применять `FLANK_PROTECTION_REDUCTION = 0.75` без необходимости формального назначения роли. Простое изменение в [`combat.py`](backend/engine/combat.py), один геометрический расчёт угла по bearing.

**2. Дальность связи — деградация по расстоянию**
Сейчас [`comms.py`](backend/engine/comms.py) деградирует связь только от подавления. Реальность: единица без радиорелейной поддержки за 5–10 км от штаба = деградация. Добавить: расчёт дистанции до ближайшей HQ/родительской единицы; `comms_range_m` в возможностях единицы; `relay_unit_types` (командный пункт, штаб) удваивают диапазон. Влияет на [`comms.py`](backend/engine/comms.py) и [`structures.py`](backend/engine/structures.py).

**3. Наблюдатель → Цикл огневой миссии (ФО→Огонь)**
Формальная цепочка: ОП/разведка «обозначает» цель — артиллерия стреляет по запросу с корректировкой. Новый тип задания `designate_target` для OP-единиц создаёт `marked_target` в `unit.capabilities`. `adjust_fire` — заказ корректировки по результату залпа. Это замыкает настоящую петлю «разведка-поражение». Затрагивает: [`order.py`](backend/schemas/order.py), [`combat.py`](backend/engine/combat.py), парсер приказов.

**4. Перекрытие огнём / Рубеж перекрытия (Overwatch)**
Тип задания `overwatch {sector_bearing, sector_width_deg}`: единица ждёт, а при обнаружении врага в секторе автоматически открывает огонь одновременно с движением дружественной единицы. Реализует реальную тактику «прыжком». Новое поле в [`tick.py`](backend/engine/tick.py) перед шагом движения.

**5. Топливо и расход ресурсов при движении**
Колёсная и гусеничная техника потребляет `fuel` (как `ammo`) при движении. `fuel < 0.2` — скорость, как в пешем режиме; `fuel = 0` — стоп. Логистические единицы заправляют топливо (аналогично боеприпасам). Минимальные изменения в [`movement.py`](backend/engine/movement.py) и [`resupply.py`](backend/engine/resupply.py).

**6. Дезактивация (РХБЗ)**
Химическое облако есть, деконтаминации нет. Новый тип задания `decontaminate`: инженерная/специальная единица устраняет химическое заражение единицы или зоны. Простое дополнение к [`engineering.py`](backend/engine/engineering.py) и [`map_objects.py`](backend/engine/map_objects.py).

## II. Командный цикл (C2)

**7. ФРАГО — Изменение задачи в полёте**
Сейчас: отменить + переиздать. Нужно: `frago` — изменяет текущий приказ (новый целевой рубеж, смена скорости, дополнительная задача) без полного сброса. Единица продолжает движение к ближайшей контрольной точке, потом переключается на новую цель. Парсер получает классификацию `frago`, движок патч-применяет `current_task`. Затрагивает: [`order_parser.py`](backend/services/order_parser.py), [`tick.py`](backend/engine/tick.py).

**8. Каскадирование замысла (Intent Cascade)**
Когда комроты отдаёт приказ взводу — командир взвода должен *автоматически* нарезать задачи отделениям. Добавить: при получении приказа HQ-единицей автоматически генерировать подзадачи для подчинённых (по иерархии CoC), с логикой нарезки по типу задачи. Первая итерация — простые правила: роте атака → взводам фланг/штурм/огневой, без LLM.

**9. Предупредительный приказ (ВАРНО)**
`order_type = "warno"`: сигнализирует единицам готовиться к определённому типу задачи. Движок переводит единицу в состояние `ready` (заняла позицию, пополнилась). Не назначает цель. Отличается от `observe` — это подготовка, а не наблюдение.

**10. Контроль объектов / Захват рубежа**
Сейчас победа оценивается LLM. Добавить: тип карточного объекта `objective` (точка/полигон) со стороной-контролёром. Единица на объекте > N тиков → объект переходит под контроль. SITREP включает статус объектов. Движок проверяет победные условия детерминированно по количеству захваченных объектов.

## III. UX для учений / Контроль тренировки

**11. Просмотр после действия (ААР / Replay)**
События хранятся append-only — инфраструктура есть. Добавить: эндпоинт `GET /sessions/{id}/replay?from_tick=&to_tick=` возвращает снимки состояния. Фронтенд: ползунок временно́й шкалы, треки движения единиц, переключатель между позициями по тикам. Это давно отложено (`AGENTS.MD Phase 10`), сейчас правильный момент.

**12. Инъекция трения (Trainer Friction Tool)**
Командование игры должно иметь возможность вводить неожиданные события: поломка техники (`unit.move_speed_mps = 0` на N тиков), ошибка координат (единица перемещается в случайную точку), отказ связи (`unit.comms_status = offline`), выход из строя командира (удаление HQ). Новый эндпоинт `POST /admin/sessions/{id}/inject-event`, UI в панели администратора.

**13. Скриптованное поведение Красных (Scripted Red Mode)**
Дополнение к LLM-управляемому Red AI: опциональный «скриптованный режим» — Red действует по заранее заданной последовательности (тик X: атака, тик Y: отход, тик Z: удержание). Скрипт редактируется в JSON-редакторе в панели Red AI. Для воспроизводимых учений с фиксированными вводными.

**14. Шаблон ОПОРД (OPORD Builder)**
В конструкторе сценариев — структурированный редактор боевого приказа по стандарту: Обстановка / Задача / Выполнение / Тыл / Управление и связь. Генерирует текст брифинга для `scenario.description` и заполняет `objectives/environment`. Адресован тренерам без технических навыков.

## IV. Авиация и специальные операции

**15. Площадка приземления (LZ/PZ) и риск высадки**
Сейчас вертолёты сажают войска в любом месте. Добавить: тип объекта `landing_zone` (нейтральный/красный/синий). Операции `air_assault`/`casevac` требуют указания LZ. Если LZ не подавлена — высадка проходит, если подавлена — вертолёт уязвим. Добавить `AVIATION_LZ_RISK` модификатор в движок.

**16. Противовоздушная оборона (ПВО)**
Вертолёты/БПЛА — уязвимы для ПВО. Добавить: типы единиц `manpads_team`, `sam_section`; в движке — проверка авиационных единиц в диапазоне ПВО при движении; `air_defense_range_m` в возможностях. Без этого авиация слишком сильна.

## V. Адаптивное самообучение из игровых сессий

**17. Дообучение `order_phrasebook.toml` и `FIELD_MANUAL.md` на реальных данных**

### Идея

Каждая проведённая игра — это размеченный датасет: текст приказа → классификация парсера → тип задачи → фактическое выполнение → результат. Анализируя эти данные, система может итеративно улучшать как парсер приказов, так и тактическую доктрину — без ручного редактирования.

### Сигналы для обучения

| Сигнал | Источник | Значение |
|---|---|---|
| Приказ отменён/переиздан в течение 1–2 тиков | `Order` table, `status=cancelled` + повтор | Указывает на ошибку парсинга или непонимание |
| LLM-классификация расходится с keyword-парсером | `order_parser.py` log | Новый паттерн, достойный добавления в фразеологизм |
| Задача выполнена без изменений = успех | `order.status=completed` | Подтверждённый рабочий паттерн |
| Отладочный лог указывает на модель "full" при уверенности <0.65 | `debug_log.txt` | Потенциально слабое место keyword-парсера |
| Приказ содержит неизвестный топоним/ориентир | `location_resolver.py` falls back to None | Пробел в доктрине ориентирования |

### Архитектура: `backend/services/learning/`

```
backend/services/learning/
├── __init__.py
├── session_analyzer.py   — анализирует завершённую сессию, извлекает пары (текст→результат)
├── phrasebook_miner.py   — предлагает новые [[case]] записи для order_phrasebook.toml
├── doctrine_reviewer.py  — выявляет пробелы доктрины, предлагает дополнения FIELD_MANUAL.md
└── proposal_store.py     — хранит предложения в БД до ручного одобрения (таблица LearningProposal)
```

### Процесс (human-in-the-loop)

1. **Сбор** — по окончании сессии или по запросу: `POST /api/admin/sessions/{id}/analyze-learning` запускает `session_analyzer.py`. Анализируются все приказы сессии, события, результаты.
2. **Минирование** — `phrasebook_miner.py` извлекает:
   - Новые устойчивые фразы из успешных LLM-парсов (которых нет в keyword-парсере) → предлагает новую строку в `[lexicon.classification]` или новый `[[case]]`
   - Фразы из отменённых приказов, которые были неверно классифицированы → предлагает исправление существующего `[[case]]`
3. **Доктрина** — `doctrine_reviewer.py` анализирует:
   - Какие типы задач командиры давали чаще всего для данного типа сценария
   - Какие ситуации порождали нестандартные формулировки (многоглагольные, смешанный RU/EN)
   - Предлагает абзацы дополнений в соответствующие разделы `FIELD_MANUAL.md` (Appendix F Topic Snippets)
4. **Ревью** — `GET /api/admin/learning/proposals` возвращает список предложений с обоснованием, примерами, уверенностью
5. **Применение** — `POST /api/admin/learning/apply` с телом `{proposal_ids: [...]}` применяет одобренные предложения: дописывает TOML-строки в phrasebook, дописывает доктринные сниппеты в FIELD_MANUAL.md

### Модель данных: `LearningProposal`

```
LearningProposal
  id              UUID  PK
  session_id      UUID  FK → Session
  proposal_type   ENUM('phrasebook_case', 'phrasebook_lexicon', 'doctrine_snippet')
  target_file     VARCHAR  -- 'order_phrasebook.toml' или 'FIELD_MANUAL.md'
  target_section  VARCHAR  -- имя раздела/lexicon-ключа / DOCTRINE:TOPIC
  proposed_text   TEXT     -- готовый TOML-блок или Markdown-абзац
  rationale       TEXT     -- почему предложено: N успехов, M отмен, паттерн
  source_order_ids UUID[]  -- ссылки на исходные приказы
  confidence      FLOAT    -- 0.0–1.0
  status          ENUM('pending', 'approved', 'rejected', 'applied')
  created_at      TIMESTAMP
  applied_at      TIMESTAMP  nullable
```

### Что НЕ делает система автоматически

- **Не перезаписывает файлы без одобрения** — только предлагает, решение за человеком.
- **Не дообучает веса LLM** — только пополняет keyword-парсер и few-shot примеры (которые идут в промпт, [`prompts/order_parser.py`](backend/prompts/order_parser.py)).
- **Не накапливает бесконечно** — предложения с confidence < 0.4 или основывающиеся на < 3 примерах отклоняются автоматически.
